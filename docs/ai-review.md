# AI レビュー機能 設計ドキュメント

## コンセプト

「変更に強い設計をコーディング段階で担保する」というプロダクトのコアバリューを実現するAI機能。

設計書（design_guides）をルールの基準として、フォルダ（ソースコード）をそのままGeminiに渡してレビューさせる。AIとの議論を通じて設計書を育て、再評価することで設計の質を継続的に高めるループを作る。

---

## AI実装方針

### 入力
- フォルダ内容（ソースコードそのまま）
- 設計書（Markdown）

ノード/エッジへの変換は行わない。Gemini（Vertex AI）の大きなコンテキストウィンドウを活かしてソースコードを直接渡す。

### 出力形式
Function Toolsを使って構造化JSONとして受け取る。

```json
{
  "overall_score": 72,
  "summary": "全体的にDDDの方向性は合っているが...",
  "design_guide_feedbacks": [
    {
      "id": "dg-1",
      "severity": "high",
      "title": "依存方向のルールが未定義",
      "description": "設計書に層間の依存方向が明記されていない",
      "suggestion": "domain層からinfrastructure層への依存を禁止するルールを追記してください"
    }
  ],
  "code_feedbacks": [
    {
      "id": "cf-1",
      "severity": "high",
      "title": "循環依存の検出",
      "description": "domain/user.go が infrastructure 層に直接依存しています",
      "file_paths": ["domain/user.go", "infrastructure/user_repo.go"],
      "node_ids": ["1", "3"],
      "edge_ids": ["2"],
      "suggestion": "Repositoryインターフェースを経由してください"
    }
  ]
}
```

### 技術スタック
- Vertex AI（Gemini） — IDEA.mdに明記、GCPインフラと統一

## レビューの2種類

### 1. 設計書へのフィードバック
設計書自体が曖昧・不足していないかを指摘する。

例:
- 「依存方向のルールが明記されていない」
- 「レイヤー定義が抽象的すぎる」

### 2. コードへのフィードバック
設計書のルールに対してコードが違反している箇所を指摘する。各指摘には関連するノード/エッジIDが紐付けられる。

例:
- 「domain層がinfrastructure層に直接依存している」
- 「1クラスへの依存が異常に多い（神クラス）」

---

## 修正パターン

カードごとの議論を経て、以下の3パターンのいずれかに確定する。

| パターン | 内容 |
|---------|------|
| **設計書を更新** | 指摘を踏まえて設計書のcontentに追記・修正する |
| **コード修正** | AIがコードの修正案を提示する |
| **両方** | 設計書の更新とコード修正案の両方を行う |

---

## レビューループ

```
① フォルダ + 設計書 → Geminiがレビュー
        ↓
② フィードバックカード一覧が生成される
        ↓
③ カードごとに会話 → 3パターンのいずれかに確定
   （設計書はこのタイミングで随時更新される）
        ↓
④ 全カード（または任意のタイミング）で「再評価」
        ↓
⑤ 更新された設計書 + 同じフォルダ → 再度Geminiレビュー
        ↓
⑥ 新しいカード一覧（スコアの変化で改善度がわかる）
```

設計書が育つたびにレビューの精度も上がる。

## UI設計

### ワークスペース通常時（右パネル）

```
┌──────────────────────┬────────────────┐
│                      │ AIレビュー     │
│  キャンバス          ├────────────────┤
│                      │ 🔴 循環依存    │
│                      │ 🔴 神クラス    │
│                      │ 🟡 命名規則    │
│                      │ ✅ 解決済み    │
│                      │                │
│                      │ [全画面で開く] │
└──────────────────────┴────────────────┘
```

- カード一覧のインデックスとして機能する
- カードをクリック → キャンバス上の関連ノード/エッジが強調（他はグレーアウト）
- 選択されたカード内に「チャットで議論する →」ボタンが出る
- そのボタンで全画面モーダルへ遷移

### 全画面モーダル

```
┌────────────────────────────────────────────────────┐
│  [設計書を更新 □ 0件]  [コード修正 □ 0件]  [両方 □ 0件]  │
├──────────────────────────┬─────────────────────────┤
│                          │                         │
│  フィードバックカード     │  チャット               │
│  （severity順）          │  （カード選択時に表示）  │
│                          │                         │
│  🔴 循環依存             │  AI: この循環依存は...   │
│  🔴 神クラス             │  You: じゃあこうすると？ │
│  🟡 命名規則             │  AI: そうですね...       │
│                          │                         │
└──────────────────────────┴─────────────────────────┘
```

### 確定アニメーション

1. チャットで議論 → 3パターンのいずれかを選択
2. カードの色が変わる（パターンに応じた色）
3. カードがふわっと浮いて上の対応する箱に向かって飛ぶ
4. 箱に収まって件数が +1、箱がちょっと揺れる

### 箱の操作

- 箱をホバー → 中に吸い込まれたカードが展開して見える
- カードをクリック → 「確定を取り消す」が出る
- 取り消すとカードが箱から飛び出してカード一覧に戻る

### 再評価

全カードが箱に収まった（or 任意のタイミング）で「再評価」ボタンが光る。

## カードのデータ構造（フロントエンド）

```typescript
type FeedbackCard = {
  id: string;
  type: 'design_guide' | 'code';
  severity: 'high' | 'medium' | 'low';
  title: string;
  description: string;
  suggestion: string;

  // コードフィードバックのみ
  filePaths?: string[];
  nodeIds?: string[];
  edgeIds?: string[];

  // 確定状態
  resolution?: 'update_design_guide' | 'fix_code' | 'both';
  resolved: boolean;

  // チャット履歴
  chatHistory: {
    role: 'user' | 'ai';
    content: string;
    timestamp: Date;
  }[];
};
```
